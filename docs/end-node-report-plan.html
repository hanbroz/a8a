1요약 (Executive Summary)

End 노드를 더블클릭하면 열리는 새 설정 모달을 만들고, 그 안에서 캔버스 실행 종료 시점에 자동으로 HTML 또는 Markdown + Mermaid 형식의 실행 이력 리포트를 파일로 저장하는 기능을 추가합니다.

  • EndNodeModal — 신규 컴포넌트
  • 리포트 형식: 없음 / HTML / MARKDOWN+MERMAID 라디오 선택
  • 저장 경로: 폴더 브라우저로 선택 (기본 ~/Downloads)
  • 파일명: 기본 템플릿 {env}_{ws}_{project}_{yyyyMMddHHmmss}, 사용자 편집 가능
  • 대상 모듈: 체크리스트로 선택 (기본 = 전체. Start/End 제외)
  • End 노드 실행 시점에 캡처한 실행 로그를 토대로 리포트 생성·저장

2사용자가 명시한 요구사항

  1. End 노드를 더블클릭하면 설정 창이 열린다 (현재 EndNodeModal이 없으므로 신규 작성 필요)
  2. Start / End는 프로젝트마다 독립적으로 존재 (이미 그렇게 동작 — 변경 없음)
  3. 리포트 형식 옵션: 없음 / HTML / MARKDOWN+MERMAID — 단일 선택
  4. 리포트 생성 선택 시 저장 경로(폴더 브라우저, 기본 Downloads), 파일명(기본 템플릿), 출력 저장 모듈(기본 전체, Start/End 제외) 입력
  5. API 리포트 — 상세도 최우선. Request / Response의 JSON은 가독성 높게 포맷팅.
  6. HTML 리포트 — JSON 뷰어 3종 (UI / Tree / JSON) 탭 전환. UI view는 각 JSON 항목을 타입 아이콘 + 라벨 + 값 카드 블록으로 표시 (Postman 응답 뷰어와 유사).

3결정이 필요한 항목

파일명 기본 템플릿을 어떻게 조합할까요? "캔버스 이름 : 환경 + 워크스페이스명 + 프로젝트명 + yyyyMMddHHmmss" 의 정확한 형태?
A. 언더바 결합: {환경}_{워크스페이스}_{프로젝트}_{20260526151230} 예시: DEV_플랫폼개발_PSS연동_20260526151230.html
B. 하이픈 결합: {환경}-{워크스페이스}-{프로젝트}-{20260526151230}
C. 다른 구분자: 사용자가 직접 명시
리포트는 언제 생성·저장됩니까?
A. 캔버스 실행이 End 노드에 도달하면 자동 — 별도 버튼 없음. 사용자 확인 없이 바로 저장. 에러 시 콘솔/로그 패널에 알림.
B. End 노드 모달에 "리포트 생성" 버튼 — 직전 실행 이력 기반. 자동 저장 없음.
C. 두 가지 모두 지원 — 자동 + 수동.
동일 파일명이 이미 존재할 때?
A. 덮어쓰기 — 기본 템플릿에 타임스탬프가 있어 충돌 가능성 낮음. 사용자가 같은 이름 명시 시 그 의도를 존중.
B. 자동 (1), (2) 붙임
C. 덮어쓰기 확인 다이얼로그
각 모듈에 대해 리포트에 포함할 정보 범위?
A. 풀 디테일 (추천) — 입력 JSON, 출력 JSON, 상태(success/error/skip), 소요시간, API의 경우 method/URL/요청헤더/요청바디/응답상태/응답바디 전체
B. 간단 — 상태 + 입출력 JSON 만
C. 사용자가 모듈별로 선택 — 체크박스로 더 세분화
Markdown+Mermaid 리포트의 다이어그램 형태?
A. flowchart LR — 좌→우 흐름. 각 노드에 상태 색상(녹색=성공, 빨강=에러, 회색=skip). 모듈명 표시.
B. sequenceDiagram — 시간 순서 + 메시지 형태. API 호출이 많을 때 적합.
C. 둘 다 포함 — flowchart + sequenceDiagram 동시.
민감 정보(Authorization 헤더, 환경변수 값 등) 처리?
A. 그대로 출력 (추천) — 디버깅 가치 우선. 사용자가 공유 시 직접 마스킹.
B. Authorization / api-key / token 등 마스킹 — *** 처리.
C. 토글 옵션 — End 노드 설정에서 켜고 끄기.
선택되지 않은 모듈도 다이어그램에는 표시되어야 하나요?
A. 다이어그램은 전체 노드 표시, 본문 섹션은 선택된 것만 — 흐름은 한눈에, 디테일은 필요한 것만.
B. 다이어그램·본문 모두 선택된 것만
HTML 리포트 — 3-탭 JSON 뷰어의 기본 활성 탭은?
A. UI (블록 카드) — 가장 가독성 높음. 스크린샷이 보여준 형식.
B. Tree — 계층 구조 한눈에. 개발자 친화적.
C. JSON — raw 텍스트. 복사·검색에 최적.
3-탭 뷰어를 적용할 영역은?
A. 모든 JSON (Request body · Response body · INPUT · OUTPUT) — 일관성 우선.
B. Request / Response 만 — API 강조. INPUT/OUTPUT은 raw JSON.
큰 JSON(예: 응답 1MB↑) 렌더링 정책?
A. UI/Tree는 100KB 이하만 자동 렌더, 그 이상은 JSON 탭만 기본 활성 + 안내 메시지 — 브라우저 성능 보호.
B. 무제한 — 사용자 책임
C. 1MB↑은 truncate — "(2,345줄 생략)" 표시.
