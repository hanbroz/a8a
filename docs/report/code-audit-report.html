#1 COMPILE SELECT 노드 — 타입 가드가 실제 요소 타입의 서브타입이 아님 SelectNodeModal.tsx:179
문제
사용자 정의 타입 가드 node is {id, parts}의 단언 타입이 원래 파라미터 타입 SelectJsonNode | undefined의 서브타입이 아니라 컴파일 에러(TS2677). 동시에 node가 undefined일 수 있다는 에러(TS18048)도 발생.
원인
TypeScript 규칙상 x is T의 T는 원래 타입에 대입 가능(서브타입)해야 함. 코드가 실제 타입 대신 SelectJsonNode의 일부 속성만 가진 구조체를 단언해 위반.
수정 전
.filter((node): node is { id: string; parts: JsonPathPart[] } => !!node)
수정 후
.filter((node): node is SelectJsonNode => !!node)
영향
한 줄 수정으로 179·180 두 에러 동시 해소. 이후 .parts 접근도 타입 안전해짐.

#3 CRITICAL · 데이터 DB 저장 시 fsync 부재 → 크래시 시 DB 전체 유실 가능 db.ts:save()
문제
writeFileSync + renameSync는 원자성은 보장하지만 내구성은 보장하지 않음. rename은 디스크에 반영됐는데 데이터 블록은 미반영된 채 전원 차단·크래시가 나면 a8a.db가 0바이트/잘린 상태로 남아 전체 워크스페이스·프로젝트·노드 데이터가 유실될 수 있음.
수정 전
function save(): void {
  const path = dbPath()
  const tmpPath = `${path}.tmp`
  writeFileSync(tmpPath, Buffer.from(db.export()))
  renameSync(tmpPath, path)
}
수정 후
function save(): void {
  const path = dbPath()
  const tmpPath = `${path}.tmp`
  const fd = openSync(tmpPath, 'w')
  try {
    writeSync(fd, Buffer.from(db.export()))
    fsyncSync(fd)          // ← 디스크에 강제 반영 후 rename
  } finally {
    closeSync(fd)
  }
  renameSync(tmpPath, path)
}
영향
크래시 시점에 관계없이 항상 온전한 DB 보장. 저위험 변경.

#5 HIGH · 정합성 프로젝트 전환 시 실행 상태(로그·상태 배지·분기) 미초기화 App.tsx:893

#6 HIGH · 안정성 http:fetch — 타임아웃 없음 + IPC 경계 밖으로 throw ipc.ts:67
문제
① 타임아웃이 없어 응답 없는 엔드포인트에 무한 대기. ② fetch 실패(DNS·연결 거부·잘못된 URL) 시 핸들러가 catch하지 않아 렌더러에 불투명한 "remote method" 에러로 전파(미처리 rejection 소지).
수정
AbortController 30초 타임아웃 + try/catch로 실패 시 { ok:false, status:0, statusText:<메시지> } 반환. 두 호출부(App.tsx:1305·1799)가 모두 !res.ok를 검사하므로 완전 호환.
판단
SSRF/사설 IP 차단은 적용하지 않음 — 이 앱은 사용자가 localhost 포함 임의 엔드포인트를 호출하는 API 테스트 도구이므로 차단 시 핵심 기능이 깨짐.
